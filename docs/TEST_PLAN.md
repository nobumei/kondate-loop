# TEST_PLAN.md — 手動テスト手順

## スモークテスト(5分・毎変更後に必須)
1. index.html を開く → 下部ナビに4タブ(在庫/献立/買い物/レシート)が表示され、コンソールエラーなし。設定タブは下部ナビに無く、ヘッダー☰→「⚙️ 設定」で開く
2. 在庫: FAB+ → 「鶏もも肉/300g/冷蔵/期限=明日」保存 → 期限が近い食材に「あと1日」で出る
3. 献立: マイクボタンの下の「次へ ▶」→ 確認画面→「🎲 この内容で提案」→ 3品出る。鶏もも系レシピが上位に来やすい(期限ボーナス)
4. 「足りない食材を買い物リストへ」→ 買い物タブに遷移し追加されている
5. 買い物: 1品チェック → 「購入済みを在庫へ」→ 在庫タブに増えている
6. レシート: 画像を1枚アップ → サムネイル表示 → タップで拡大
7. 読み上げ: 買い物タブの「🔊 読み上げ」ボタン → 音声が出る(ミュート解除確認)。献立タブに読み上げボタンが無いこと(v1.9で廃止)
8. リロード → 全データが残っている(献立タブは前回の提案結果画面から始まる)

## 回帰テスト(リリース前)
- 在庫の編集・削除 / 期限なし食材が一覧末尾に来る
- レシピ追加(材料改行パース)・編集・削除、seedレシピに削除ボタンが無い
- お気に入り/頻度/評価の保存、「あまり」設定が提案に出にくい
- 「この献立を作った」→ 最終調理日更新 → 直後の再提案で同じ主菜が出にくい
- 買い物曜日トグル、エクスポート→全削除→インポートで復元
- スマホ実機: 375px幅でレイアウト崩れなし、カメラ起動、ホーム画面追加

## 家族同期テスト(v1.1)
1. 端末A: 設定 > 家族で同期 > あいことば生成 > 同期を開始 → 🟢になる
2. 端末B: 同じあいことばを入力 > 同期を開始 → 確認ダイアログ → 端末Aの在庫が現れる
3. 端末Bで買い物リストにチェック → 端末Aをバックグラウンド→復帰 → 反映されている
4. 機内モードで在庫追加 → 🔴表示 → 通信復帰後、次の変更操作で🟢に戻る
5. 同期を解除 → ローカルデータは残る/以後通信しない(DevToolsのNetworkで確認)
6. エクスポートJSONに householdId が含まれていないこと
7. レシート画像が他端末に現れないこと(仕様どおり)

## v1.7 機能テスト(献立提案への要望反映・調理モード)
1. 献立タブの「今日の要望」を空のまま「献立を提案する」→ 従来どおり即座に提案される(DevToolsのNetworkでGeminiへの通信が発生していないこと)
2. 設定タブでGemini APIキー未設定のまま要望を入力して提案→「Gemini APIキーが未設定のため通常の提案を使用します。」がトースト表示され、従来ロジックで提案される(コンソールエラーは出ない)
3. APIキー設定済みで要望を入力して提案→ Gemini呼び出し中は「⏳ AIが考え中…」表示→ 提案結果の下に「💡 AIの提案理由: ...」が表示される。まとめ提案でも同様
4. (開発確認)Geminiが存在しないレシピidを返した場合、そのidは無視され、従来ロジックで該当カテゴリが補完されること(不正idでエラーにならない)
5. Gemini APIエラー(429/401/403/その他)時は該当メッセージがトースト表示され、従来ロジックにフォールバックする
6. 要望・AIの提案理由はページをリロードすると消えている(stateに保存されないこと)
7. 音声入力非対応ブラウザ(または`window.SpeechRecognition`/`webkitSpeechRecognition`が存在しない環境)では🎤ボタンが表示されないこと。対応環境では🎤タップで音声認識が開始し、認識結果が要望欄に入ること
8. 作り方が登録されているレシピの詳細で「🔊 調理モード」ボタンが表示され、未登録レシピでは表示されないこと
9. 調理モードを開くと「工程 1/N」と現在の工程が表示され、読み上げが開始される。「次へ」「前へ」「もう一回」のタップがすべて即座に効くこと(音声認識の状態に関わらず)
10. 最終工程の「次へ」後は「完了」表示になり、ボタンが「閉じる」に変わる。閉じるとモーダルが閉じる
11. 調理モード表示中に背景タップ/「閉じる」ボタン/最終工程後の「閉じる」のいずれで閉じても、`speechSynthesis.cancel()`と音声認識の停止が必ず実行されること(閉じた後に音声が流れ続けない)
12. 音声認識対応環境では、読み上げ完了後に自動で待受が始まり、「つぎ/次/OK/オッケー」で次へ、「もう一回/もういちど」で再読み上げ、「もどる/前」で前へ進む。それ以外の発話(誤認識)では何も起きない

1. 買い物タブ「📰 チラシから特売を取り込む」→ 画像を選択→ Gemini抽出結果の確認リストが開く(チェックON状態)。編集・チェック解除ができ、「チェックした◯件を取り込む」→ 特売リストに反映される。チラシ画像自体がlocalStorage(`kondate-loop-v1`)に保存されていないこと(DevToolsでstate.dealsにdataUrl等の画像データが含まれないことを確認)

12. 読み取り→在庫に追加を実行していない(itemsが無い)レシートは集計の月選択・金額に含まれないこと。集計対象がない月は「集計できるレシートがありません」の注記が出る

7. 旧データ(`state.settings.shoppingLog`に日付が入った状態)をインポート→ エラーなく読み込め、エクスポートJSONにも`shoppingLog`がそのまま残っていること(データ自体は非破壊)
8. `renderAll()`がエラー無く実行される(在庫追加・削除など通常操作後も画面が正しく再描画される)
9. 作り方が登録済みのレシピで「🔊 調理モード」を開く→ 材料がある場合は最初に「材料」カードが表示され、読み上げも「材料は、〇〇大さじ2、△△300グラム…です。」のように分量付きで再生される。「次へ」で工程1に進み、工程1で「前へ」を押すと材料カードに戻れる(それより前には戻れない)
10. 材料が空のレシピ(手動でings未入力のレシピを作成)で調理モードを開く→ 材料カードは出ず工程1から始まる
11. 豚の生姜焼きの材料表示・調理モードで「しょうが」が「10g」(曖昧な「1かけ」ではない)として表示・読み上げられること

## v1.11 機能テスト(レシピOCR取り込み・検索リンク直リンク化)
1. Gemini APIキー未設定で献立タブ「📷 本・画面からレシピを取り込む」→ 画像を選択→「先に設定タブでGemini APIキーを登録してください。」トースト+設定タブへ自動遷移(レシートOCRと同じ挙動)
2. APIキー設定済みで料理本の写真を選択→ スピナー表示→ 抽出結果の確認モーダルが開き、レシピごとに名前(編集可)・区分(主菜/副菜/汁物セレクト)・調理時間・材料(name/qty/unit、行の追加・削除可)・作り方(1行1手順、行の追加・削除可)がチェックON状態で表示される
3. 画像に複数のレシピが写っている場合、確認モーダルにレシピごとのカードが複数並ぶこと
4. 1件だけチェックを残して他を外し、残した1件の名前・材料・作り方を編集して「チェックした1件をレシピに追加」→ レシピ一覧・件数表示に反映され、チェックを外したレシピは追加されないこと。追加されたレシピは`seed:false`で削除ボタンが表示され、献立提案の候補にも出ること
5. 材料が1つも読み取れなかったレシピは自動的にスキップされ、「材料が読み取れなかった◯件はスキップしました。」がトースト表示されること。名前はあるが材料が空の場合も同様にスキップされる
6. 0件抽出または解析失敗時は「読み取れませんでした。」の空表示になり、追加ボタンが表示されないこと
7. 確認モーダルで全レシピのチェックを外した状態で追加ボタンを押すと「追加できるレシピがありません。」のトーストが出て、`state.recipes`が変化せずモーダルも閉じないこと
8. APIエラー時(429/401/403/その他)は該当メッセージがトースト表示される(レシートOCRと同じ文言)。エクスポートJSONにGemini APIキーが含まれないこと(既存キーと同じ端末ローカル方針)
9. レシピ詳細(タップ→作り方)の「もっと詳しく」リンクが `https://www.kurashiru.com/search?query=<レシピ名>` と `https://delishkitchen.tv/search?q=<レシピ名>` になっていること(材料は付与しない、target=_blank・rel=noopener を維持)
