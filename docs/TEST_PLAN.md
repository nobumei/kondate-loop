# TEST_PLAN.md — 手動テスト手順

## スモークテスト(5分・毎変更後に必須)
1. index.html を開く → 5タブが表示され、コンソールエラーなし
2. 在庫: FAB+ → 「鶏もも肉/300g/冷蔵/期限=明日」保存 → 期限が近い食材に「あと1日」で出る
3. 献立: 提案 → 3品出る。鶏もも系レシピが上位に来やすい(期限ボーナス)
4. 「足りない食材を買い物リストへ」→ 買い物タブに遷移し追加されている
5. 買い物: 1品チェック → 「購入済みを在庫へ」→ 在庫タブに増えている
6. レシート: 画像を1枚アップ → サムネイル表示 → タップで拡大
7. 読み上げ: 献立読み上げボタン → 音声が出る(ミュート解除確認)
8. リロード → 全データが残っている

## 回帰テスト(リリース前)
- 在庫の編集・削除 / 期限なし食材が一覧末尾に来る
- レシピ追加(材料改行パース)・編集・削除、seedレシピに削除ボタンが無い
- お気に入り/頻度/評価の保存、「あまり」設定が提案に出にくい
- 「この献立を作った」→ 最終調理日更新 → 直後の再提案で同じ主菜が出にくい
- 買い物曜日トグル、買い物記録+1、エクスポート→全削除→インポートで復元
- スマホ実機: 375px幅でレイアウト崩れなし、カメラ起動、ホーム画面追加

## 家族同期テスト(v1.1)
1. 端末A: 設定 > 家族で同期 > あいことば生成 > 同期を開始 → 🟢になる
2. 端末B: 同じあいことばを入力 > 同期を開始 → 確認ダイアログ → 端末Aの在庫が現れる
3. 端末Bで買い物リストにチェック → 端末Aをバックグラウンド→復帰 → 反映されている
4. 機内モードで在庫追加 → 🔴表示 → 通信復帰後、次の変更操作で🟢に戻る
5. 同期を解除 → ローカルデータは残る/以後通信しない(DevToolsのNetworkで確認)
6. エクスポートJSONに householdId が含まれていないこと
7. レシート画像が他端末に現れないこと(仕様どおり)

## v1.4 レシートOCR(Gemini)テスト
1. キー未設定でレシートの「🔍 読み取り」→「先に設定タブでGemini APIキーを登録してください。」トースト+設定タブへ自動遷移
2. 設定タブでAPIキーを保存→レシートタブで「🔍 読み取り」→スピナー表示→確認モーダルが開き、抽出品目がチェックON状態で並ぶ
3. 確認モーダルで1品だけチェックを残し、name/qty/unit/location/expiryを編集→「チェックした1品を在庫に追加」→在庫タブに反映され、チェックを外した品は追加されない
4. 0品抽出または解析失敗時は「読み取れませんでした。手動で追加してください。」と表示され、追加ボタンが出ない
5. APIエラー時: 429→「無料枠の上限です。1分待ってから再試行してください。」、401/403→「APIキーを確認してください。」がトースト表示される
6. エクスポートJSONに Gemini APIキーが含まれていないこと

4. localStorageを空にして初回起動→使い方チュートリアルが自動表示される→「はじめる」で閉じ、`kondate-loop-tutorial-seen` が保存される→リロードしても再表示されない

8. レシートOCRの確認モーダルで、抽出名が辞書の別名だった場合に正規名へ初期変換されていること(モーダル上で編集可)

## v1.6 機能テスト(日数×食事選択・在庫チップ手動補正・事前準備集約・献立カレンダー)
1. 献立タブの「まとめ献立と仕込み」で日数を3、朝と夜にチェックを入れて「まとめて提案」→ 3日分×朝/夜の献立が表示され、朝は汁物+主菜1品の軽量構成になる
2. 単発提案・まとめ提案どちらの材料表示も【◯食材名】(緑)/【✗食材名】(赤)のチップになっている
3. 在庫がある食材のチップ【◯...】をタップ→【✗...】に切り替わる→「まとめ買いリストへ」を押すとその食材が買い物リストに追加される(逆に✗→◯にすると追加されない)。この補正はリロードすると消える(保存されない)
4. 複数日にまたがる食材(例: にんじん)が事前準備リストに合計量+使い先の内訳(「◯本は今日の◯◯、◯本は明日の◯◯で使用」のような1行)で表示され、offset2日目以降の分は冷凍案内が付く
5. 開発者ツールでlocalStorageの`kondate-loop-v1`を旧v1.5形式(`plan.days[].ids`配列、`history`キーなし)に書き換えてリロード→ エラーなく起動し、`state.plan.days[0].meals.夜`に旧献立が入り、`state.history`が`[]`で補完されている
6. 献立タブの「この献立を作った」または👍/👎フィードバックを行うと、献立カレンダーの当日セルにドットが付く→ 日付をタップするとその日の献立名一覧(朝/昼/夜)が表示される
7. 献立カレンダーの「前月」「次月」で表示月が切り替わる
8. 履歴に登録済みのレシピを削除した状態でその日をタップ→「(削除済み)」と表示されエラーが出ない

## v1.7 機能テスト(献立提案への要望反映・調理モード)
1. 献立タブの「今日の要望」を空のまま「献立を提案する」→ 従来どおり即座に提案される(DevToolsのNetworkでGeminiへの通信が発生していないこと)
2. 設定タブでGemini APIキー未設定のまま要望を入力して提案→「Gemini APIキーが未設定のため通常の提案を使用します。」がトースト表示され、従来ロジックで提案される(コンソールエラーは出ない)
3. APIキー設定済みで要望を入力して提案→ Gemini呼び出し中は「⏳ AIが考え中…」表示→ 提案結果の下に「💡 AIの提案理由: ...」が表示される。まとめ提案でも同様
4. (開発確認)Geminiが存在しないレシピidを返した場合、そのidは無視され、従来ロジックで該当カテゴリが補完されること(不正idでエラーにならない)
5. Gemini APIエラー(429/401/403/その他)時は該当メッセージがトースト表示され、従来ロジックにフォールバックする
6. 要望・AIの提案理由はページをリロードすると消えている(stateに保存されないこと)
7. 音声入力非対応ブラウザ(または`window.SpeechRecognition`/`webkitSpeechRecognition`が存在しない環境)では🎤ボタンが表示されないこと。対応環境では🎤タップで音声認識が開始し、認識結果が要望欄に入ること
8. 作り方が登録されているレシピの詳細で「🔊 調理モード」ボタンが表示され、未登録レシピでは表示されないこと
9. 調理モードを開くと「工程 1/N」と現在の工程が表示され、読み上げが開始される。「次へ」「前へ」「もう一回」のタップがすべて即座に効くこと(音声認識の状態に関わらず)
10. 最終工程の「次へ」後は「完了」表示になり、ボタンが「閉じる」に変わる。閉じるとモーダルが閉じる
11. 調理モード表示中に背景タップ/「閉じる」ボタン/最終工程後の「閉じる」のいずれで閉じても、`speechSynthesis.cancel()`と音声認識の停止が必ず実行されること(閉じた後に音声が流れ続けない)
12. 音声認識対応環境では、読み上げ完了後に自動で待受が始まり、「つぎ/次/OK/オッケー」で次へ、「もう一回/もういちど」で再読み上げ、「もどる/前」で前へ進む。それ以外の発話(誤認識)では何も起きない

## v1.8 機能テスト(チラシOCR特売+スコア加点・レシート集計)
1. 買い物タブ「📰 チラシから特売を取り込む」→ 画像を選択→ Gemini抽出結果の確認リストが開く(チェックON状態)。編集・チェック解除ができ、「チェックした◯件を取り込む」→ 特売リストに反映される。チラシ画像自体がlocalStorage(`kondate-loop-v1`)に保存されていないこと(DevToolsでstate.dealsにdataUrl等の画像データが含まれないことを確認)
2. 特売リストの品目名と、在庫に近い名前の買い物リスト品目に「特売 ¥◯◯」バッジが付く(normalizeName一致。例: 特売名「たまねぎ」→買い物リスト「玉ねぎ」でも一致)
3. 献立提案で特売食材を含むレシピのスコアが底上げされ、再提案で選ばれやすくなる(同一レシピをscoreRecipeで比較すると特売ありの方が高スコア)
4. 提案画面の【◯/✗】チップで、✗かつ特売中の食材のチップに「特売」の表示が追加される。在庫があり◯表示の食材には付かない
5. 特売の期限(validUntil)が今日より前のものは特売リスト・バッジ・スコア加点のいずれにも出てこない(非表示なだけでstate.dealsのデータ自体は残る)
6. 特売リスト先頭の「🧹 期限切れを削除」→ 確認ダイアログ→ 期限切れの特売のみ削除される(在庫・買い物リスト・レシピ等は変更されない)。期限切れが0件のときはボタンが表示されない
7. Gemini APIキー未設定でチラシ取り込みを実行→「先に設定タブでGemini APIキーを登録してください。」トースト+設定タブへ自動遷移。APIエラー時(429/401/403)は該当メッセージがトースト表示される
8. 旧形式データ(`deals`キーなし)を開いてもエラーが出ず、`state.deals`が`[]`で補完される
9. レシートの「🔍 読み取り」→ 確認モーダルに各品目のジャンル(野菜/肉/魚/乳製品/主食/調味料/飲料/菓子/日用品/その他)が表示され、食品ジャンルはチェックON、日用品/その他は既定OFFになっている
10. 確認モーダルで「在庫に追加」を実行→ チェックした品だけが在庫に追加され、チェックの有無に関わらず全抽出品目+レシート合計金額(total)がそのレシートのitems/totalとして保存される
11. レシートタブ「📊 集計」で月を選択→ その月の月間総額(レシートにtotalがあれば優先、なければitems合計)・ジャンル別内訳(金額降順の横バー表示)・レシート枚数が表示される。複数月にレシートがある場合、月選択に両方が表示され切り替えられる
12. 読み取り→在庫に追加を実行していない(itemsが無い)レシートは集計の月選択・金額に含まれないこと。集計対象がない月は「集計できるレシートがありません」の注記が出る
